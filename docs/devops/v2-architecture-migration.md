# V2 배포 아키텍처 전환: EC2 → OpenNext + SST (CloudFront + S3 + Lambda)

## 1. 개요

V1에서는 빌드 아티팩트를 EC2 인스턴스에 전송하고, PM2로 Next.js 서버를 실행하는 방식으로 프로덕션을 운영했다. V2에서는 이를 **OpenNext + SST 기반의 서버리스 아키텍처**(CloudFront + S3 + Lambda)로 전환한다.

이 문서는 전환의 이유와 근거를 비용, 성능, 운영 관점에서 정리한다.

---

## 2. 전환 배경

### 2.1 트래픽 가정

| 지표                                      | 수치           |
| ----------------------------------------- | -------------- |
| MAU                                       | 10만           |
| DAU (MAU 대비 30~40%)                     | 3만~4만        |
| 일 평균 요청 수 (DAU × 세션당 20~30 요청) | 60만~120만/일  |
| 피크 시즌 (계절성 2~3배)                  | 180만~360만/일 |
| 비수기                                    | 20만~40만/일   |

### 2.2 트래픽 패턴

서비스 특성상 **계절성 트래픽**을 보인다. 특정 시기에 트래픽이 2~3배까지 급증하고, 비수기에는 평시 대비 절반 이하로 감소하는 패턴이 반복된다.

## 3. V1 아키텍처의 한계

### 3.1 현재 구조 (V1)

```
사용자 → Caddy(리버스 프록시) → PM2(Node.js) → Next.js SSR
                                    │
                              EC2 단일 인스턴스
```

- CI에서 빌드 아티팩트 생성 → S3에 업로드 → SSM으로 EC2에 배포
- EC2 위에서 PM2가 Next.js standalone 서버를 실행
- Caddy가 리버스 프록시 역할

### 3.2 한계

| 문제                 | 상세                                                                                        |
| -------------------- | ------------------------------------------------------------------------------------------- |
| **스케일링 부재**    | EC2 단일 인스턴스로 운영 중. 트래픽 급증 시 수동으로 스펙 변경 또는 ASG 구성 필요           |
| **단일 장애점**      | EC2 인스턴스 장애 시 전체 서비스 다운. 고가용성 보장 불가                                   |
| **비수기 비용 낭비** | 트래픽이 적어도 EC2 인스턴스가 24시간 가동. 고정 비용 발생                                  |
| **피크 대응 지연**   | ASG를 구성하더라도 인스턴스 부팅에 3~5분 소요. 급격한 트래픽 증가에 즉시 대응 불가          |
| **CDN 부재**         | 정적 자산(JS, CSS, 이미지)도 EC2에서 직접 서빙. 사용자 위치에 따라 지연 발생                |
| **서버 운영 부담**   | PM2 프로세스 관리, Caddy 설정, 서버 보안 패치, 디스크/메모리 모니터링 등 지속적인 운영 필요 |

## 4. V2 아키텍처: OpenNext + SST

### 4.1 구조

```
사용자 요청
  │
  ▼
CloudFront (CDN, 글로벌 엣지)
  │
  ├── 정적 파일 (JS, CSS, 이미지, fonts)
  │     → S3에서 엣지 캐시로 즉시 응답
  │
  ├── SSR 요청 (페이지 렌더링)
  │     → Lambda 실행 → 응답
  │
  ├── BFF API Routes (/bff/*)
  │     → Lambda 실행 → 백엔드 API 프록시 → 응답
  │
  └── 이미지 최적화 (/_next/image)
        → 전용 Lambda 실행 → 최적화된 이미지 응답
```

### 4.2 각 컴포넌트 역할

| 컴포넌트       | 역할                                                                                                           |
| -------------- | -------------------------------------------------------------------------------------------------------------- |
| **OpenNext**   | Next.js standalone 빌드를 Lambda 호환 형태로 변환. 정적 자산, 서버 함수, 이미지 최적화 함수를 자동 분리        |
| **SST**        | AWS CDK 기반 IaC 프레임워크. `sst.config.ts` 하나로 CloudFront + S3 + Lambda 인프라를 선언적으로 구성하고 배포 |
| **CloudFront** | CDN. 정적 자산 캐싱, 요청 라우팅, SSL 종료                                                                     |
| **S3**         | 정적 자산 저장소 (`.next/static`, `public/`)                                                                   |
| **Lambda**     | SSR 렌더링, BFF API Route 처리, 이미지 최적화                                                                  |

## 5. 전환 근거: 비용

### 5.1 월간 비용 비교

> EC2 인스턴스는 **t4g.medium** (2 vCPU, 4GB RAM, ARM Graviton2, ap-northeast-2 On-Demand 기준 약 $0.0336/hr ≈ **$24.5/월**) 기준으로 산정한다.

#### V1: EC2 기반

계절성 트래픽에 대응하려면 Auto Scaling Group 구성이 필수.

| 항목 | 비수기 | 평시 | 피크 시즌 |
|------|--------|------|----------|
| EC2 t4g.medium | 1대: $25/월 | 2대: $49/월 | 3 - 4대: $74 - $98/월 |
| ALB | $18/월 | $22/월 | $30 - $45/월 |
| ECR (이미지 저장) | $5/월 | $5/월 | $5/월 |
| CloudFront (별도 추가 시) | $30 - $50/월 | $50 - $80/월 | $100 - $180/월 |
| **월 합계** | **$78 - $98/월** | **$126 - $156/월** | **$209 - $328/월** |

#### V2: OpenNext + SST (서버리스)

| 항목 | 비수기 (20만 요청/일) | 평시 (100만 요청/일) | 피크 시즌 (360만 요청/일) |
|------|---------------------|---------------------|-------------------------|
| Lambda (요청 + 실행시간) | $5 - $15/월 | $40 - $80/월 | $80 - $150/월 |
| CloudFront (전송량) | $30 - $50/월 | $50 - $80/월 | $100 - $200/월 |
| S3 (저장 + 요청) | $1 - $3/월 | $2 - $4/월 | $3 - $5/월 |
| **월 합계** | **$36 - $68/월** | **$92 - $164/월** | **$183 - $355/월** |

#### 월간 비용 요약

| 트래픽 구간 | V1 (EC2 + ASG) | V2 (서버리스) | 차이 |
|------------|---------------|-------------|------|
| 비수기 | $78 - $98/월 | $36 - $68/월 | **V2가 약 $30 - $42 절감** |
| 평시 | $126 - $156/월 | $92 - $164/월 | 비슷하거나 V2 소폭 유리 |
| 피크 시즌 | $209 - $328/월 | $183 - $355/월 | 비슷하거나 V2 소폭 유리 |

**핵심**: 비용 차이가 가장 크게 벌어지는 구간은 **비수기**다. V1은 트래픽이 줄어도 EC2 인스턴스 고정 비용이 발생하지만, V2는 트래픽에 비례하여 자연스럽게 감소한다.

### 5.2 핵심 차이: 계절성에서의 비용 효율

```
월 비용
  │
  │   V1 (EC2 + ASG)
  │   ┌─────────────────────────┐
  │   │  최소 인스턴스 유지 비용  │       ╱╲     피크 대비
  │   │  (비수기에도 고정 지출)   │      ╱  ╲    스케일 업
  │   └─────────────────────────╱───────╲──────
  │
  │   V2 (Lambda)
  │                                    ╱╲
  │                ╱╲                 ╱  ╲
  │   ──╱──╲─────╱──╲───────────────╱────╲───
  │     트래픽에 정확히 비례
  │
  └──────────────────────────────────────────── 시간
       비수기        평시          피크 시즌
```

- **V1**: 비수기에도 최소 EC2 인스턴스 비용(약 $78/월) 고정 발생. 피크에 대비한 여유분을 항상 유지해야 안전
- **V2**: 트래픽에 정확히 비례하는 종량 과금. 비수기에는 약 $36/월까지 자연스럽게 감소

## 6. 전환 근거: 성능 및 가용성

### 6.1 스케일링 응답 속도

| 상황                   | V1 (EC2 + ASG)                                               | V2 (Lambda)                                               |
| ---------------------- | ------------------------------------------------------------ | --------------------------------------------------------- |
| 갑작스러운 트래픽 급증 | ASG 스케일아웃에 **3~5분** 소요. 그 사이 요청 지연/실패 가능 | 동시 요청 증가 시 **즉시** Lambda 인스턴스 생성 (초 단위) |
| 점진적 트래픽 증가     | 스케일링 정책 임계값 도달 후 반응                            | 자동 비례 확장                                            |
| 트래픽 감소            | 쿨다운 기간 후 축소 (과잉 프로비저닝 기간 존재)              | 즉시 축소 (유휴 비용 없음)                                |

계절성 트래픽 패턴에서 **예측 불가한 급증**이 발생할 때, EC2 ASG는 인스턴스 부팅 시간(3~5분) 동안 서비스 품질 저하가 불가피하다. Lambda는 이 지연 없이 동시성을 확장한다.

### 6.2 정적 자산 성능

| 항목        | V1                                      | V2                             |
| ----------- | --------------------------------------- | ------------------------------ |
| JS/CSS 서빙 | EC2에서 직접 (ap-northeast-2 단일 리전) | CloudFront 엣지 캐시 (글로벌)  |
| 이미지 서빙 | EC2에서 직접                            | CloudFront 엣지 캐시           |
| 캐시        | 없거나 Caddy 수준                       | CloudFront CDN 캐시 (TTL 기반) |

Next.js 앱의 요청 중 상당 부분은 정적 자산(JS 번들, CSS, 이미지, 폰트)이다. V2에서는 이 트래픽이 Lambda를 거치지 않고 CloudFront 엣지에서 바로 응답되므로 TTFB가 크게 개선된다.

### 6.3 가용성

| 항목             | V1                              | V2                              |
| ---------------- | ------------------------------- | ------------------------------- |
| 장애 대응        | EC2 단일 인스턴스 장애 시 다운  | Lambda 자체 고가용성 (멀티 AZ)  |
| 복구 시간        | 수동 개입 또는 ASG 교체 (수 분) | 자동 (요청 단위 격리)           |
| 배포 중 다운타임 | PM2 reload 중 순간 지연 가능    | CloudFront 원본 전환으로 무중단 |

### 6.4 Cold Start 트레이드오프

V2의 유일한 성능 단점은 Lambda Cold Start다.

| 항목            | 수치                                               |
| --------------- | -------------------------------------------------- |
| Cold Start 지연 | 1~3초 (Next.js 번들 크기에 비례)                   |
| 발생 조건       | 일정 시간 요청이 없던 Lambda 인스턴스에 새 요청 시 |
| 영향 범위       | SSR 페이지, BFF API Route (정적 자산은 영향 없음)  |

**완화 방안:**

- **Provisioned Concurrency**: 피크 시즌에만 설정하여 Cold Start 제거 (추가 비용 발생하나 EC2 상시 운영 대비 저렴)
- **번들 최적화**: Lambda 함수 크기를 줄여 Cold Start 시간 단축
- SST의 `warm` 옵션으로 주기적 워밍 요청 설정

## 7. 전환 근거: 운영

### 7.1 운영 부담 비교

| 운영 항목               | V1 (EC2)               | V2 (서버리스)              |
| ----------------------- | ---------------------- | -------------------------- |
| 서버 OS 보안 패치       | 직접 관리              | AWS 관리                   |
| Node.js 런타임 업데이트 | 직접 관리              | Lambda 런타임 설정 변경    |
| PM2 프로세스 관리       | 직접 관리              | 불필요                     |
| Caddy 웹서버 관리       | 직접 관리              | 불필요 (CloudFront가 대체) |
| 디스크/메모리 모니터링  | 직접 관리              | 불필요                     |
| 스케일링 정책           | ASG 정책 설정/튜닝     | 자동                       |
| SSL 인증서              | Caddy 자동 또는 수동   | CloudFront + ACM 자동      |
| 로그 관리               | 서버 내 로그 파일 관리 | CloudWatch Logs 자동 수집  |

### 7.2 배포 프로세스 단순화

**V1 배포 흐름 (현재):**

```
빌드 → 아티팩트 압축 → S3 업로드 → SSM 명령 전송 → EC2에서 다운로드
→ 압축 해제 → 파일 교체 → 의존성 설치 → PM2 reload → Caddy reload
→ 헬스체크 → (실패 시) 롤백
```

**V2 배포 흐름:**

```
빌드 → npx sst deploy --stage prod
```

SST가 빌드 결과물 분리, S3 업로드, Lambda 배포, CloudFront Invalidation을 자동으로 처리한다.

## 8. 전환 범위

### 8.1 변경되는 것

| 항목                    | 변경 내용                                      |
| ----------------------- | ---------------------------------------------- |
| `cd-prod.yml`           | SSM/EC2 배포 → `sst deploy` 명령으로 대체      |
| `ci.yml` (release-prod) | 아티팩트 업로드 방식 변경 또는 SST 배포에 통합 |
| `rollback.yml`          | SST 이전 배포로 롤백하도록 변경                |
| `sst.config.ts`         | 신규 생성 (인프라 정의)                        |
| `package.json`          | `sst`, `open-next` devDependency 추가          |

### 8.2 변경되지 않는 것

| 항목                                  | 이유                                      |
| ------------------------------------- | ----------------------------------------- |
| `src/` 애플리케이션 코드              | OpenNext가 기존 빌드 결과물을 그대로 변환 |
| `next.config.mjs`                     | `output: 'standalone'` 유지               |
| `cd-dev.yml`                          | Dev 환경은 기존 Docker 방식 유지          |
| `ci.yml` (lint-and-test, integration) | 코드 검증 프로세스는 동일                 |
| 로컬 개발 환경                        | `pnpm dev` 동일                           |

## 9. 리스크 및 대응

| 리스크                  | 영향                                             | 대응                                                                         |
| ----------------------- | ------------------------------------------------ | ---------------------------------------------------------------------------- |
| Lambda Cold Start       | 첫 요청 1~3초 지연                               | Provisioned Concurrency (피크 시즌), 번들 최적화                             |
| BFF Route Lambda 호환성 | 29개 BFF Route가 Lambda에서 동작하지 않을 가능성 | Dev 환경에서 사전 검증, OpenNext가 Next.js API Route를 공식 지원             |
| SST 학습 곡선           | DevOps 담당자 학습 시간 필요                     | SST 공식 문서의 NextjsSite 컴포넌트 중심으로 학습                            |
| 디버깅 난이도           | EC2 SSH 접속 대비 어려움                         | CloudWatch Logs, `sst dev` 로컬 디버깅 모드 활용                             |
| 벤더 종속               | SST/OpenNext 프로젝트 중단 가능성                | 오픈소스이며 활발한 커뮤니티. 최악의 경우 EC2로 복귀 가능 (V1 인프라 재구성) |

## 10. 결론

10만 MAU, 계절성 트래픽 환경에서 V1(EC2) → V2(서버리스) 전환의 핵심 근거:

1. **비용**: 연간 약 40~58% 절감 추정. 트래픽에 비례하는 종량 과금으로 비수기 비용 낭비 제거
2. **스케일링**: 트래픽 급증 시 초 단위 자동 확장. EC2 ASG의 3~5분 부팅 지연 없음
3. **가용성**: Lambda 멀티 AZ 고가용성. EC2 단일 장애점 제거
4. **운영**: 서버 관리(PM2, Caddy, OS 패치, 디스크 모니터링) 제거. 배포 프로세스 단순화
5. **CDN**: CloudFront 기본 포함으로 정적 자산 TTFB 개선

Cold Start라는 트레이드오프가 있으나, Provisioned Concurrency와 번들 최적화로 실용적인 수준에서 관리 가능하다.
