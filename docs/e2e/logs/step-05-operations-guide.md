# Step 05 - 운영 가이드/안정화 규칙 문서화

## 1) 배경 및 목표

### 배경

- Step 04까지 테스트/CI는 구현됐지만, 팀 기준 문서가 없으면 작성 방식이 분산된다.

## 6) 이슈 및 대응

### 이슈 A: 운영 문서 범위가 너무 넓어지면 유지보수 부담 증가

- 현상: 하나의 문서에 구현 상세/기획 논의까지 넣으면 빠르게 낡음
- 원인: 운영 가이드와 작업 로그의 역할이 섞일 수 있음
- 대응: 가이드는 실행 규칙 중심으로 고정하고, 상세 시행착오는 Step 로그/트러블슈팅 인덱스로 분리

## 7) 트러블슈팅 타임라인

1. 증상:

- 문서화 항목이 많아지면서 가이드가 장황해질 위험

2. 확인한 가설:

- 가설 A: 가이드에 모든 히스토리를 넣으면 읽기성이 떨어진다.
- 가설 B: 운영 규칙과 시행착오를 분리하면 유지보수가 쉬워진다.

3. 시도한 조치:

- 가이드는 "현재 유효한 규칙"만 작성
- 실패 사례/고충은 logs + troubleshooting에만 유지

4. 최종 원인:

- 문서 목적이 서로 다른데 한 파일에 합치면 과밀화됨

5. 최종 해결:

- `guide.md`(운영 규칙) + `logs/`(상세 이력) + `troubleshooting.md`(인덱스) 3층 구조 확정

## 8) 고충/의사결정 메모

- 당시 가장 헷갈렸던 지점: 가이드와 로그의 경계를 어디까지 둘지.
- 버린 접근: Step 로그 내용을 운영 가이드에 중복 복사하는 방식.
- 채택한 접근: 운영 가이드는 안정 규칙만, 상세 이력은 로그로 분리.
- 다음 단계에서 같은 문제를 피하는 방법: 변경 작업 시 먼저 "규칙 변경인지, 이력 추가인지"를 구분하고 문서 위치를 결정.
