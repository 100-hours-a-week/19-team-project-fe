# 2026-02-22 개발일지

## 주제

BFF 타임아웃/SSR fallback 도입 및 장기 작업(이력서 파싱, 레포트 생성) 30초 정책 적용

---

## 1) 문제 배경

### 실제 장애 시나리오

- 백엔드 서버가 응답하지 않는 상황에서 SSR이 BFF를 호출하면 `fetch`가 무기한 대기할 수 있음
- 결과적으로 Lambda(또는 서버리스 실행 환경) 최대 실행 시간에 걸려 페이지 자체가 `504`로 실패
- 사용자는 "데이터가 잠깐 안 뜨는" 문제가 아니라 "페이지가 아예 안 뜨는" 문제를 겪음

### 기존 코드의 구조적 한계

- BFF route 대부분이 `fetch(buildApiUrl(...))`를 직접 호출
- 공통 타임아웃/로그/에러 표준화 레이어 부재
- SSR 일부는 실패를 흡수했지만(`catch(() => [])`) degraded 상태를 명시적으로 전달하지 않음

---

## 2) 목표

- BFF upstream 호출에 공통 타임아웃 적용
- 타임아웃 발생 시 무기한 대기 대신 즉시 표준 에러(`504`, `UPSTREAM_TIMEOUT`) 반환
- SSR에서 fallback 데이터를 사용해 페이지 렌더 자체는 유지
- 장애 원인을 추적할 수 있는 구조화 로그 추가
- 장기 작업 API만 예외적으로 타임아웃을 늘려 정상 성공률 보장

---

## 3) 설계 고민과 선택

### 고민 A: 모든 API에 동일 타임아웃을 줄 것인가?

- 동일값은 단순하지만 API 성격이 다름
- 추천 API(비핵심)와 파싱/리포트 생성(장기 작업)을 같은 값으로 두면 둘 중 하나가 항상 손해

선택:

- 프로파일 기반 기본값 + 엔드포인트별 override
- `optional(2s)`, `default(3.5s)`, `critical(5s)`
- 장기 작업은 route/server 레벨에서 `timeoutMs: 30000` 명시

### 고민 B: timeout 처리 위치를 어디에 둘 것인가?

- 각 route에 직접 `AbortController`를 넣으면 중복 증가
- 공통 wrapper로 모으면 일관성/유지보수성 개선

선택:

- 공통 wrapper `fetchBffUpstream` 도입
- route는 이 wrapper만 호출

### 고민 C: SSR fallback은 어디서 할 것인가?

- BFF에서만 처리하면 UI가 degraded 상태를 인식하지 못함
- SSR에서도 `degraded` 플래그를 전달해야 사용자에게 맥락 제공 가능

선택:

- SSR server component에서 `try/catch` + `FALLBACK_DATA` + `degraded` 전달

### 고민 D: 장기 작업 타임아웃 값

- 5초는 파싱/리포트 생성에 너무 짧음
- 30초는 길지만 실행환경 상한(예: Lambda 30초)과 거의 맞닿아 있어 운영 판단 필요

선택:

- 요구사항에 맞춰 30초 적용
- 향후 비동기 작업 전환(idempotency key + task polling) 권장

## 4) 구현 상세

## 4-1. 공통 타임아웃 유틸 추가

파일: `src/shared/api/server/withTimeout.server.ts`

핵심 포인트:

- `AbortSignal.timeout` 사용 가능 시 우선 사용
- 미지원 환경 fallback으로 `AbortController + setTimeout`
- 외부 signal과 timeout signal 동시 지원 (`AbortSignal.any` 또는 수동 결합)
- 타임아웃일 때만 `RequestTimeoutError`로 정규화

```ts
export class RequestTimeoutError extends Error {
  readonly timeoutMs: number;

  constructor(timeoutMs: number) {
    super(`REQUEST_TIMEOUT_${timeoutMs}MS`);
    this.name = 'RequestTimeoutError';
    this.timeoutMs = timeoutMs;
  }
}

export async function withTimeout<T>(
  operation: (signal: AbortSignal) => Promise<T>,
  options: WithTimeoutOptions,
): Promise<T> {
  const timeoutSignal = createTimeoutSignal(options.timeoutMs);
  const signal = options.signal ? combineSignals([options.signal, timeoutSignal]) : timeoutSignal;

  try {
    return await operation(signal);
  } catch (error) {
    if (timeoutSignal.aborted && isAbortError(error)) {
      throw new RequestTimeoutError(options.timeoutMs);
    }
    throw error;
  }
}
```

또한 FSD 규칙 준수를 위해 public export 추가:
파일: `src/shared/api/server/index.ts`

```ts
export { withTimeout, RequestTimeoutError } from './withTimeout.server';
```

## 4-2. BFF 공통 fetch 래퍼 도입

파일: `src/app/bff/_lib/fetchUpstream.ts`

역할:

- upstream fetch 공통 타임아웃 적용
- 프로파일 기반 timeout 계산
- timeout 발생 시 표준 504 응답 생성
- 장애/지연 로그 표준 태깅

```ts
type TimeoutProfile = 'optional' | 'default' | 'critical';

const TIMEOUT_BY_PROFILE: Record<TimeoutProfile, number> = {
  optional: 2000,
  default: 3500,
  critical: 5000,
};

function resolveTimeoutProfile(upstreamPath: string, method: string, override?: TimeoutProfile) {
  if (override) return override;
  if (upstreamPath.startsWith('/api/v1/experts/recommendations')) return 'optional';
  if (upstreamPath.startsWith('/api/v1/resumes/tasks')) return 'critical';
  if (upstreamPath.startsWith('/api/v2/reports')) return method === 'GET' ? 'critical' : 'default';
  return 'default';
}
```

```ts
function createTimeoutResponse(timeoutMs: number, requestId: string): Response {
  return Response.json(
    {
      code: 'UPSTREAM_TIMEOUT',
      message: 'UPSTREAM_TIMEOUT',
      data: null,
      degraded: true,
      requestId,
      timeoutMs,
    },
    { status: 504 },
  );
}
```

```ts
if (error instanceof RequestTimeoutError) {
  const requestId = crypto.randomUUID();
  console.error('[BFF_UPSTREAM_TIMEOUT]', {
    event: 'bff_upstream_timeout',
    bffPath: bffPath ?? upstreamPath,
    upstreamPath,
    method,
    timeoutProfile: resolvedTimeoutProfile,
    timeoutMs,
    requestId,
  });

  return createTimeoutResponse(timeoutMs, requestId);
}
```

장애 로그 포맷:

- timeout: `event: bff_upstream_timeout`
- upstream 5xx: `event: bff_upstream_http_error`

## 4-4. SSR fallback/degraded 구조 도입

파일: `src/widgets/home/ui/ExpertRecommendationsServer.tsx`

이전:

- 실패 시 단순 `[]` 반환
- 실패 맥락(degraded) UI 전달 없음

이후:

- `FALLBACK_DATA` 상수화
- `try/catch`로 실패 시 degraded=true
- 구조화 로그 출력

```ts
const FALLBACK_DATA: ExpertRecommendationsResponse = {
  user_id: 0,
  recommendations: [],
  total_count: 0,
  evaluation: {},
};

let data = FALLBACK_DATA;
let degraded = false;

try {
  data = await apiFetch<ExpertRecommendationsResponse>(url, { ... });
} catch (error) {
  degraded = true;
  console.error('[SSR_EXPERT_RECOMMENDATIONS_FALLBACK]', {
    event: 'ssr_expert_recommendations_fallback',
    degraded,
    reason: error instanceof Error ? error.message : 'unknown',
    path: '/bff/experts/recommendations',
  });
}

return <ExpertRecommendations recommendations={data.recommendations} degraded={degraded} />;
```

파일: `src/widgets/home/ui/ExpertRecommendations.tsx`

```tsx
type ExpertRecommendationsProps = {
  recommendations: ExpertRecommendation[];
  degraded?: boolean;
};
```

```tsx
{
  degraded ? <div className="...">추천 서비스 지연으로 일부 데이터가 누락될 수 있어요.</div> : null;
}
```

결과:

- 백엔드가 느리거나 장애여도 홈은 렌더 유지
- 사용자는 단순 빈 화면이 아니라 "현재 degraded 상태"를 인지 가능

## 6) 트레이드오프 정리

### 장점

- 무기한 대기 제거
- 페이지 전체 504 위험 감소
- 장애 원인 파악 가능한 로그 확보
- SSR degraded/fallback로 UX 손실 완화

### 비용/리스크

- 타임아웃이 너무 짧으면 정상 요청도 실패 처리(조기 중단)
- 타임아웃 후 사용자 재시도 시 중복 요청 가능성
- 엔드포인트별 timeout 정책 관리 복잡도 증가

### 운영 관점 결론

- "짧은 기본 타임아웃 + 장기 작업 예외"가 현재 구조에서 가장 현실적
- 장기적으로는 비동기 작업 전환(task polling) + idempotency key가 더 안전

---

## 7) idempotency key 관련 메모

- 프론트 단독으로는 불완전
- 백엔드가 키를 저장/중복판단/재반환을 지원해야 진짜 중복 방지 가능
- 이번 변경은 timeout/fallback 안정화가 1차 목표였고, idempotency는 후속 과제로 남김
